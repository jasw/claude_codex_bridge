# Source Ownership Manifest

Date: 2026-06-07

## Problem

CCB generated provider memory currently mixes several ownership domains:

- CCB runtime coordination rules generated by the renderer.
- Shared project memory from `.ccb/ccb_memory.md`.
- Provider user memory from source homes such as `~/.claude/CLAUDE.md` or
  Codex source-home `AGENTS.md`.
- Provider-native project memory such as project `CLAUDE.md` or `AGENTS.md`.
- Agent-private CCB memory.
- Provider-native rules or instruction directories.

When the same provider also loads some of those sources natively, the generated
CCB bundle can duplicate memory. Text-level deduplication is unsafe because two
similar rules may have different ownership, and user-authored text must not be
silently removed because it resembles generated CCB text.

## Manifest Shape

The durable contract should be a source ownership manifest:

```text
source kind -> native loaded? -> CCB bundle included? -> filter policy
```

The manifest is provider-specific, but source kinds should be shared across
providers where possible.

## Source Kinds

| Source kind | Owner | Intended role |
| :--- | :--- | :--- |
| `runtime_coordination_rules` | CCB renderer | Canonical managed ask and task coordination rules. |
| `ccb_shared` | CCB project | User-editable shared team memory at `.ccb/ccb_memory.md`. |
| `provider_user_memory` | Provider account/user | Inherited source-home memory, after CCB-owned install text is stripped. |
| `provider_native_project` | Provider/project | Project memory that the provider can discover natively. |
| `agent_private` | CCB project | Per-agent memory under `.ccb/agents/<agent>/memory.md`. |
| `rules_dir` | Provider account/user | Provider-native rules directory; not a CCB bundle input. |

## Target Policy

| Provider | Source kind | Native loaded? | CCB bundle included? | Filter policy |
| :--- | :--- | :--- | :--- | :--- |
| Claude | `runtime_coordination_rules` | No | Generated once | None |
| Claude | `ccb_shared` | No | Yes | None |
| Claude | `provider_user_memory` | Account/user home | Yes | Strip CCB install blocks and legacy CCB ask blocks |
| Claude | `provider_native_project` | Claude native project memory | No | None |
| Claude | `agent_private` | No | Yes | None |
| Claude | `rules_dir` | Provider native or unstable | No | Never include |
| Codex | `runtime_coordination_rules` | No | Generated once | None |
| Codex | `ccb_shared` | No | Yes | None |
| Codex | `provider_user_memory` | Source `CODEX_HOME` or account home | Yes | Strip CCB install blocks and legacy CCB ask blocks |
| Codex | `provider_native_project` | Codex native project `AGENTS.md` | No | None |
| Codex | `agent_private` | No | Yes | None |
| OpenCode | `runtime_coordination_rules` | No | Generated once | None |
| OpenCode | `ccb_shared` | No | Yes | None |
| OpenCode | `provider_user_memory` | Project/user config dependent | Conditional | Strip CCB install blocks when inherited |
| OpenCode | `provider_native_project` | OpenCode native project `AGENTS.md` discovery | No | None |
| OpenCode | `agent_private` | No | Yes | None |
| OpenCode | `rules_dir` | No stable CCB ownership | No | Never include |
| Gemini | `runtime_coordination_rules` | No | Generated once | None |
| Gemini | `ccb_shared` | No | Yes | None |
| Gemini | `provider_user_memory` | Source home | Yes | Strip CCB install blocks when inherited |
| Gemini | `provider_native_project` | Needs audit; context-file behavior differs | Deferred until audited | None |
| Gemini | `agent_private` | No | Yes | None |

Gemini is listed to prevent accidental omission, but its `provider_native_project`
row is deliberately deferred until the managed `contextFileName` path is audited.

The OpenCode `provider_native_project` row was audited against OpenCode 1.16.2:
OpenCode discovers project `AGENTS.md` natively while also loading configured
`instructions`, so CCB excludes project `AGENTS.md` from the generated runtime
bundle and keeps the generated bridge instruction for CCB-owned memory only.

## Filtering Rules

Filtering must be source-aware:

- Apply CCB install-block stripping only to `provider_user_memory`, because it
  removes CCB-managed installation residue from provider account memory.
- Do not filter `.ccb/ccb_memory.md`, agent-private memory, or provider-native
  project memory by similarity.
- Do not remove arbitrary user text because it resembles CCB instructions.
- Keep the renderer-owned runtime coordination section as the only source of
  current ask protocol rules.

Candidate filtered blocks:

- `CCB_CONFIG_START` through `CCB_CONFIG_END`.
- `CCB_ROLES_START` through `CCB_ROLES_END`.
- `CODEX_REVIEW_START` through `CODEX_REVIEW_END`.
- `GEMINI_INSPIRATION_START` through `GEMINI_INSPIRATION_END`.
- Install-time inline CCB config sections written to provider user memory.
- Legacy CCB ask protocol sections that were installed into provider user
  memory by older CCB versions, such as older `Codex Collaboration Rules` or
  `Gemini Collaboration Rules` sections.
- Claude route-mode files such as `~/.claude/rules/ccb-config.md` are not CCB
  bundle inputs. New route-mode installs should stop writing that file; uninstall
  and cleanup paths should continue removing known CCB-owned copies.

Each filter should report whether it changed the source in projection events or
diagnostics without emitting secret or large source content.

Filters must be conservative: strip only complete recognized block pairs or
recognized legacy section shapes. Isolated start/end markers and unrelated
user-defined markers must remain untouched.

## Implementation Notes

- Add a small policy module near project memory source loading, for example
  `lib/project_memory/policy.py`.
- Replace ad hoc flags such as provider-native project inclusion with a policy
  lookup keyed by provider and source kind.
- Keep `render_memory_bundle()` focused on rendering ordered source objects and
  the canonical runtime coordination section.
- Preserve source labels in rendered bundles and diagnostics so users can see
  why a section was included.
- Keep `inherit_memory=false` behavior provider-specific but make it disable
  the CCB-generated memory projection consistently.
- Do not read `~/.claude/rules/` or similar provider rules directories as CCB
  memory sources.
- Auto-upgrade old `.ccb/ccb_memory.md` files only when `memory.seed.json`
  proves the current content still matches an older generated seed, or when the
  current content exactly matches a known generated legacy template. Leave
  user-edited files untouched.
- Resolve Codex source-home `AGENTS.md` semantics before filtering it: identify
  whether the file is user-authored account memory, an older CCB-written file,
  or absent in normal Codex homes. Current implementation classifies it as
  `provider_user_memory` and applies conservative CCB install-block filtering.

## Acceptance Criteria

- Managed Claude bundle includes filtered provider user memory, `.ccb/ccb_memory.md`,
  and agent-private memory when present, but not project `CLAUDE.md`.
- Managed Codex bundle includes filtered provider user memory, `.ccb/ccb_memory.md`,
  and agent-private memory when present, but not project `AGENTS.md`.
- OpenCode has an explicit tested policy for project `AGENTS.md` inclusion or
  exclusion; the policy is documented in the OpenCode contract.
- Gemini has an explicit audited manifest row before any Gemini implementation
  change is made.
- Rendered bundles contain the CCB ask protocol once.
- Default newly created `.ccb/ccb_memory.md` no longer duplicates the renderer
  ask protocol.
- Existing user-edited `.ccb/ccb_memory.md` files are not overwritten.
- Tests cover source inclusion/exclusion and provider-user-memory filtering for
  Claude and Codex.
- Provider-user-memory filter tests cover complete marker pairs, isolated
  marker non-removal, unrelated marker non-removal, and legacy collaboration
  section removal.

## Required Tests

1. Codex generated memory excludes project `AGENTS.md` after the Codex policy
   is changed.
2. Claude generated memory excludes project `CLAUDE.md` while retaining filtered
   source-home `CLAUDE.md`.
3. `load_memory_sources()` excludes provider-native project memory when the
   provider policy says so.
4. Provider-user-memory filters strip known complete install marker pairs.
5. Provider-user-memory filters strip known legacy unmarked collaboration-rule
   sections from released installer paths.
6. Provider-user-memory filters preserve isolated markers and unrelated
   user-defined marker text.
7. OpenCode generated instructions contain the CCB bridge path and do not
   accidentally inline raw project `AGENTS.md` unless the audited policy says
   to include it.
8. A new default `.ccb/ccb_memory.md` no longer duplicates ask protocol text,
   while the rendered bundle still contains the renderer-owned ask protocol
   once.
9. Copy workspace and git-worktree agents still source shared memory from
   `project_root/.ccb/ccb_memory.md`.
10. `inherit_memory=false` removes or suppresses generated memory projection
    consistently for Claude, Codex, and OpenCode.
